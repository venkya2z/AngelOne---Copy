Design Principles
Separate signal type — do not overload ENTRY; use ADD_LOT (or SCALE_IN).
Same contract only — add only when token / tradingsymbol matches active_position.
Atomic state updates — update active_position only after confirmed fill.
Idempotency — dedupe by signal_id or (ts, event_type, lots).
Configurable caps — max adds per trade, max total lots, cooldown between adds.
Full audit trail — log every leg for reconciliation and debugging.

the lot size is already decided in settings as per confidence low,medium,high righ?

Yes — at initial entry, lot count comes from settings keyed by confidence, not from the Redis signal itself.

How it works today
1. Config maps confidence → lots

"confidence_num_lots": {
  "HIGH": 3,
  "MEDIUM": 2,
  "LOW": 1
},
2. Entry reason → confidence

CONFIDENCE_MAP = {
    "OPPOSITION_ENTRY": "HIGH",
    "ALIGNMENT_ENTRY": "MEDIUM",
    "CUMULATIVE_ENTRY": "LOW"
}
Manual entries (MANUAL_ENTRY) fall back to LOW (1 lot) because they aren’t in that map.

3. Risk manager applies config + funds

confidence_map = self.config.get("confidence_num_lots", {"HIGH": 5, "MEDIUM": 3, "LOW": 1})
desired_num_lots = confidence_map.get(confidence, 1)
...
for num_lots in range(desired_num_lots, 0, -1):
    total_cost = cost_per_lot * num_lots
    if total_cost <= usable:
        return num_lots
So the flow is:

ENTRY signal reason → confidence (HIGH/MEDIUM/LOW) → desired lots from config → capped by available funds
Example with your current settings and enough margin:

OPPOSITION_ENTRY → HIGH → 3 lots
ALIGNMENT_ENTRY → MEDIUM → 2 lots
CUMULATIVE_ENTRY → LOW → 1 lot
What this means for add-lot / averaging
Lot size is decided once at entry; there is no add-lot path today.
Redis ENTRY does not carry a lots field — only the reason (and thus confidence) drives sizing.
For a future ADD_LOT, you’d need to choose:
Same rule — add the same count as entry confidence (e.g. HIGH → +3), or
Fixed add — e.g. always +1 per ADD_LOT, or
Explicit in signal — lots: 1 in the Redis message.
If you want adds to follow the same confidence settings, the natural rule would be: each ADD_LOT adds confidence_num_lots[confidence] again, still subject to max_total_lots and remaining margin.
